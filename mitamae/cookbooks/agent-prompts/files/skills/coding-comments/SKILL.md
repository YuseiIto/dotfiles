# コメントについて

## 原則

- コメント = **Why・Why not**  vs ドキュメント = **What**／コード = **How**
- アノテーションコメントを活かして、そのコメントの性質を明示する
- 過剰にコメントしない。コードを読めばわかることは書かない。

## コメントのチェックリスト

- [ ] コメントの内容が、命名や型, アサーションなどの他の方法で表現できないこと
- [ ] コードとは異なる抽象度で語るコメントであること (コードより具体的:内部のトリッキーな挙動の説明, コードより抽象的: 処理の流れの中での当該操作の目的)
- [ ] 一般的な慣習に従っただけの部分に対するコメントではないこと
- [ ] ユーザとの対話で用いた ad-hoc な表現 (比喩、フェーズ番号など) が混在していないこと (→ 後から読んだ別の人をかえって混乱させることを防ぐため)
- [ ] 一般的なプラクティスから逸れる箇所ではその理由を明記していること
- [ ] 非自明な前提に基づく場合はその前提と該当箇所への影響を明記していること
- [ ] アノテーション (TODO:, FIXME: 等) にはチケット参照を必須とし、「いつ」を具体的な日付かイベント条件で明記すること（曖昧な「後で」は禁止。詳細は末尾セクション）
- [ ] 無意味なフェーズ / ステップ番号を付与していないこと
- [ ] 過剰に長大なブロックコメントやバナーコメントではなく、凝縮された簡潔なコメントであること
- [ ] 廃止予定は `@Deprecated` / `@deprecated` 等の言語機能で表現し、NOTE/WARNING コメントで代用しない
- [ ] (できるだけ) アノテーションを付与していること

## コメントに書くべきこと

| 内容タイプ | いつ書くか | 例 |
|---|---|---|
| 意図・根拠（Why） | コードから推測できない「なぜこうしたか」 | なぜリトライを3回に制限したのか |
| 設計判断・トレードオフ | 自明な手法ではなく非自明な手法を選んだ理由 | 「素直な二分探索より Boyer-Moore が実データで速かった」 |
| Why not（却下した代替案・地雷） | 読者が「もっと良い書き方」を試して壊しそうな箇所 | 「parseInt に変えるな。末尾ゴミを黙って受理しID破壊」 |
| 不変条件・事前条件 | 型では強制できない正しさの前提 | 「呼出前にロック必須」「リストは常に昇順」 |
| 単位・範囲・null意味・所有権 | 宣言だけでは曖昧な意味（型が表現できない事実） | 「timeout はミリ秒。-1 は無制限」「見つからなければ null」 |
| 副作用・結果の警告 | 呼び出し側が予期しない影響 | 「これを呼ぶと全テナントのキャッシュが消える」 |
| 外部根拠への参照 | 外部仕様の実装・既知バグの回避 | 「RFC 5322 §3.4 準拠」「JIRA-1234 参照」 「Ref: https://doc.rust-lang.org/stable/」|
| インターフェース契約 | 公開API。呼出側が本体を読まず使えるよう動作・引数・戻り値・例外を記述 | メソッドヘッダの契約記述 |


## アノテーションコメント


| マーカー | 意味 | 例|
|---|---|---|
| TODO | 将来的な実装や改善が必要な箇所。 | `// TODO: ケースが3つ以上になったらバリデーションを強化する` |
| NOTE | コード上の重要な意図や設計上の決定。| `// NOTE: この定数は正確性とパフォーマンスのバランスを実験でチューニングして決定した。` |
| FIXME | 既知の問題や不具合。修正の詳細や方向性を含めるとgood. | `// FIXME: 大量データ処理時にメモリリーク発生` |
| HACK | 既知問題への一時的回避策。多用は設計問題のシグナル | `// HACK: ライブラリの制約の回避のため直接DBアクセスを実装` |
