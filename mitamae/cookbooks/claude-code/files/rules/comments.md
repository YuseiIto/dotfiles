# コメントについて

Claude は過剰なコメントや冗長なコードを生成しやすい傾向にある。

一般的な原則として、**ドキュメントには What を コードには How を コメントには why not** を記述する。

## コメントのチェックリスト

- [ ] コードの意図を説明するためのコメントであること (コードの内容を説明するためのコメントではないこと)
- [ ] 一般的な慣習に従っただけの部分に対するコメントではないこと
- [ ] ユーザとの対話で用いた ad-hoc な表現 (比喩、フェーズ番号など) が混在していないこと (→ 後から読んだ別の人をかえって混乱させることを防ぐため)
- [ ] 一般的なプラクティスから逸れる箇所ではその理由を明記していること
- [ ] 非自明な前提に基づく場合はその前提と該当箇所への影響を明記していること
- [ ] 将来の修正を期待する場合 (TODO:, FIXME:) は、その旨といつ直すべきかを明記すること
- [ ] 無意味なフェーズ / ステップ番号を付与していないこと
- [ ] 過剰に長大なブロックコメントではなく、凝縮された簡潔なコメントであること
