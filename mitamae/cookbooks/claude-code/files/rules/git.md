# git の扱い

## コミットメッセージ

[Conventional Commits](https://www.conventionalcommits.org/) に従う。プロジェクトに明文化された規則があればそれを優先する。

使用する type:

| type | 用途 |
| --- | --- |
| feat | 新機能の追加 |
| fix | バグ修正 |
| perf | パフォーマンス改善 |
| refactor | バグ修正・機能追加を伴わないコード変更 |
| style | コードフォーマット等の修正 |
| test | テストの追加・修正 |
| docs | ドキュメントのみの変更 |
| build | ビルドシステムや外部依存に関する変更 |
| ci | CI 設定・スクリプトの変更 (例: GitHub Actions) |
| chore | その他の変更 |
| revert | 以前のコミットの取り消し |

## コミットの粒度

- 1 コミット = 1 つの論理的変更 (ファイル数ではなく目的で区切る)。レビューと revert の単位として独立していること
- 異なる type を混ぜない。途中で関係ない問題に気づいた場合も別コミットに分ける
- 各コミットは単独でビルド・テストが通る状態であること (bisect 可能性)
- 中間コミット (WIP, typo fix 等) は PR を開く前に rebase で整理する。autosquashを見越したfixup コミットの活用も有効。
- 大きな機能を実装する際は *前提となるリファクタ → 機能追加* の順で段階的に分ける

## コミット前の検証

- リポジトリで定義された lint / format / test がローカルで通過していることを確認する
- ステージング前に `git status` / `git diff` で意図しない変更が混入していないか確認する

## ステージング

意図せぬ変更の混入を防ぐため、`git add .` のような無差別ステージングは行わない。変更内容を確認し、必要なファイルのみを明示的に指定する。

## GitHubの操作
- `gh` コマンドを用いて行うのを基本とする
- 現在の環境で提供されている場合は、 GitHub MCPツールも用いてよい
- `gh api` サブコマンドはなるべく用いない. 代わりに、 `gh pr` などの適切なサブコマンドを繰り返して用いる
    - `gh api` サブコマンドは破壊的な操作が可能であり、人間の確認を要するが、安全なサブコマンドは既定で許可されているため
